## 二、格式规范

### 2.1 中英文混排

1. 英文字符和数字两侧加空格
2. 专有名词 (如变量名、函数名、命令名等) 保持英文原文，不进行翻译
3. 英文缩写第一次出现时，先给出全称，后跟括号内的缩写形式
4. 圆括号用半角 `()`，并将其视为英文字符，两侧酌情添加空格

### 2.2 标点符号

- **中文引号**：使用 `「」` (而非 `""`)
- **中文段落**：使用全角标点 `。，；？！`
- **英文段落**：使用半角标点 `, . ; ? !`

### 2.3 章节编号

```
## 1. 一级标题
### 1.1 二级标题
#### A. 三级标题 (或 **加粗文字**，上下空行)
```

**注意事项**：
- 不设五级标题
- 避免内容过少的小节，每节建议 500-700 字
- 确保每个小节内容足够充实

### 2.4 数学公式 (如需要)

- 行内公式：`$y=mx+b$`
- 行间公式：`$$E=mc^2$$` (上下空行，居中)
- 特殊符号转义：`\&`, `\%`, `\#`
- 避免公式中出现中文

### 2.5 代码块

```
- 使用三个反引号包裹
- 指定语言类型
- 前后各空一行
- 代码宽度不超过 70 列
- Stata 换行符采用 `///`
```

### 2.6 图片占位

```markdown
<img src="https://fig-lianxh.oss-cn-shenzhen.aliyuncs.com/todo_xxx.png" width="700" alt="图片描述">
```

**说明**：使用 `todo_` 前缀标记待替换的图片，后续由作者自行截图替换。

## 三、内容风格要求

### 3.1 写作原则

- **开门见山**：第一节直接点出用户痛点，引发共鸣
- **场景驱动**：用具体场景说明问题，避免抽象描述
- **案例丰富**：每个功能都给具体例子，确保可操作性
- **循序渐进**：从基础到进阶，符合学习曲线
- **避免冗余**：聚焦核心功能，不要面面俱到
- **语言朴实**：口语化表达，多用「你」而非「用户」

### 3.2 目标读者定位

针对 **大多数小白用户**，甚至之前没有听说过这个软件的读者：
- 避免使用过于专业的术语
- 每个新概念都要解释清楚
- 提供足够的背景信息
- 步骤描述详细具体

## 四、文章结构建议

### 4.1 入门教程型 (3000-3500 字)

```
1. 引言：痛点 + 软件价值 (500字)
   - 3-4 个典型痛点场景
   - 软件如何解决这些问题
   - 软件的核心特点

2. 软件简介 (400字)
   - 是什么
   - 为什么好用
   - 下载安装

3. 基础用法 (800字)
   - 最基本的操作
   - 界面介绍
   - 3-5 个基础功能演示

4. 进阶技巧 (800字)
   - 提高效率的技巧
   - 组合使用方法
   - 常用快捷键

5. 实用场景举例 (600字)
   - 2-3 个真实应用场景
   - 对比传统方法 vs 新方法

6. 常见问题 (300字)
   - 3-5 个最常见问题及解答

7. 结语 (200字)
   - 总结软件价值
   - 给出使用建议
```

### 4.2 进阶技巧型 (3500-4000 字)

```
1. 引言：为什么需要这个功能 (500字)
   - 研究者的典型痛点
   - 这个功能的价值定位
   - 支持的场景/文件类型

2. 应用场景概览 (600字)
   - 场景 1：详细描述 + 典型需求
   - 场景 2：详细描述 + 典型需求
   - 场景 3：其他实用场景

3. 配置/使用详解 (800字)
   - 详细的配置步骤 (配图)
   - 参数说明
   - 注意事项

4. 语法/技巧详解 (700字)
   - 基础语法
   - 组合搜索
   - 精确匹配
   - 高级技巧 (简要介绍)

5. 实战演练 (800字)
   - 3-4 个手把手操作案例
   - 每个案例包含：需求 + 操作步骤 + 结果

6. 注意事项 (400字)
   - 常见陷阱
   - 解决方案
   - 最佳实践

7. 使用技巧 (300字)
   - 性能优化
   - 工具组合使用
   - 工作流建议

8. 常见问题 (300字)
   - 3-4 个核心问题

9. 结语 (200字)
   - 总结价值
   - 给出建议
```

## 五、参考文献格式 (myAPA)

### 5.1 学术文献 (需要 Google 链接)

**arXiv 论文**：
```
- 作者, 年份, 标题 (Version X). arXiv. [Link](DOI链接), [PDF](PDF链接), [Google](<Google Scholar链接>).
```

**NBER 工作论文**：
```
- 作者, 年份, 标题. National Bureau of Economic Research. [Link](DOI链接), [PDF](PDF链接), [Google](<Google Scholar链接>).
```

**正式发表论文**：
```
- 作者, 年份, 标题. 期刊名, 卷(期), 页码. [Link](DOI链接), [PDF](PDF链接), [Google](<Google Scholar链接>).
```

### 5.2 普通文献 (不需要 Google 链接)

**网站/文档/工具介绍**：
```
- 作者/机构, 年份, 标题. URL链接.
```

**示例**：
```
1. Everything 官方网站. <https://www.voidtools.com/zh-cn/>
2. VScode 官方文档. <https://code.visualstudio.com/docs>
```

### 5.3 注意事项

- `[Google](<xxx>)` 中的链接用尖括号包裹，避免特殊字符解析错误
- 无法确认的 URL 请留空，**千万不要编造链接**
- 只有学术文献才添加 `[Google](<>)` 链接

- 需要管理大量文献笔记和代码片段

**实际填写**：
```

**场景 1**：

**场景 2**：
```
[详细描述场景、文件路径、典型需求]

## 七、输出要求

### 7.1 先提供写作规划

在开始写作前，请先输出：

1. **章节结构**：列出所有章节标题
2. **字数分配**：每个章节的预计字数
3. **重点内容**：每节的核心要点
4. **图片需求**：需要多少张图、在哪些位置

**等待确认后再开始写作**

### 7.2 分段输出

- 每次输出 1-2 个完整章节
- 输出完毕后询问是否继续
- 便于及时反馈和调整

### 7.3 输出格式

- Markdown 格式
- 便于复制到 VScode 编辑器
- 可直接用于 Quarto render
- 最终发布为 GitHub Pages

### 7.4 最终检查

输出完成后，请检查：

- [ ] 字数是否符合预期 (3000-4000 字)
- [ ] 格式是否规范 (中英文、标点、章节)
- [ ] 案例是否具体可操作
- [ ] 图片占位是否合理
- [ ] 参考文献格式是否正确

## 八、迭代优化指南

### 8.1 常见修改需求

**篇幅调整**：
```
「太长了，请精简到 3500 字」
「某章节内容太少，需要扩充到 600 字」
```

**内容调整**：
```
「第 X 节缺少具体案例，请补充」
「某个概念解释不够清楚，需要展开」
```

**风格调整**：
```
「语言太正式，需要更口语化」
「某些段落过于啰嗦，请精简」
```

**结构调整**：
```
「第 X 节和第 Y 节可以合并」
「某章节顺序不合理，建议调整」
```

### 8.2 反馈方式

请按照 **「具体问题 + 期望效果」** 的方式提出修改建议，例如：

```
✅ 好的反馈：
「第 5 节的案例太抽象，请改为使用 D:\papers 这个真实路径的例子」

❌ 不好的反馈：
「第 5 节不太好」
```

## 十、使用流程

1. **复制模板**：保存本文件为 `软件推文提示词模板.md`

2. **填写信息**：
   - 第一部分：基本信息
   - 第六部分：具体写作指令

3. **提交给 AI**：
   ```
   我需要写一篇软件介绍推文，请按照以下模板帮我完成：
   
   [粘贴填好的模板内容]
   ```

4. **审阅规划**：
   - 检查章节结构是否合理
   - 确认字数分配是否恰当
   - 提出调整建议

5. **分段审阅**：
   - 每完成 1-2 个章节就审阅
   - 及时提出修改意见
   - 避免大规模返工

6. **最终检查**：
   - 统一检查格式规范
   - 替换图片占位符
   - 补充参考文献

7. **保存成果**：
   - 将成功案例保存到 `examples/` 文件夹
   - 总结经验，优化模板
